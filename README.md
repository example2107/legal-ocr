# ЮрДок — Распознавание и обезличивание юридических документов

Веб-приложение для OCR-распознавания юридических документов и автоматического обезличивания персональных данных. Пользователь загружает сканы (JPG/PNG/PDF) или DOCX, приложение распознаёт текст через Claude/OpenAI/Gemini API, находит персональные данные и позволяет заменить их на метки (А., Б., [ФИО 1] и т.д.).

## Архитектура и поток данных

### Режимы загрузки

**Изображения/PDF (главная страница):**
1. `pdfUtils.js` конвертирует PDF в base64-изображения постранично
2. `claudeApi.js → recognizeDocument()` — OCR каждой страницы отдельным запросом
3. Страницы склеиваются с маркерами `[PAGE:N]`
4. Quality check — поиск OCR-артефактов и семантических ошибок
5. Анализ ПД — структурированный JSON с persons и otherPD
6. `RichEditor.js → buildAnnotatedHtml()` — двухпроходная вставка маркеров (см. ниже)
7. Автосохранение — документ сохраняется сразу после распознавания

**PDF в проекте:**
1. Те же шаги 1–6, но с передачей `existingPD` из последнего документа проекта
2. `mergePD()` объединяет новые ПД с накопленной базой проекта
3. `assignLetters()` назначает буквы с учётом уже занятых
4. Документ сохраняется с `projectId` — не появляется в общей истории

**DOCX:**
1. `docxParser.js` читает XML напрямую через JSZip (без OCR)
2. Сразу идёт на анализ ПД (quality check пропускается)

### Views (экраны приложения)

- `VIEW_HOME` — главная: загрузка файлов, API ключ, проекты, история документов
- `VIEW_PROJECT` — внутри проекта: загрузка PDF, документы проекта, итоговый документ
- `VIEW_PROCESSING` — прогресс-бар во время распознавания
- `VIEW_RESULT` — редактор с панелью ПД и вьюером оригинала

### Состояние в App.js

Ключевые useState:
- `view` — `'home' | 'project' | 'processing' | 'result'`
- `personalData` — `{ persons: [...], otherPD: [...] }` — данные от API
- `anonymized` — `{ [id]: bool }` — какие ПД скрыты
- `editorHtml` — текущий HTML редактора
- `pdWidth`, `viewerWidth` — ширины панелей (изменяются ресайзером)
- `pdNavState` — `{ [id]: { cur, total } }` — состояние навигации по упоминаниям
- `currentProjectId` — id текущего проекта (null если не в проекте)
- `pdIdsInDoc` — Set id-шников ПД с маркерами в текущем документе (для визуализации в проекте)

Ключевые useRef (не вызывают ре-рендер):
- `editorDomRef` — прямая ссылка на DOM редактора
- `pdRef` — синхронная копия `personalData` (всегда актуальна)
- `anonRef` — синхронная копия `anonymized` (всегда актуальна)
- `undoStackRef` — стек снимков для undo/redo `[{html, pd, anon}]`
- `undoIndexRef` — текущая позиция в стеке

## Проекты

Проекты — пространства для объединения нескольких частей одного дела с общей базой персональных данных.

### Хранение (history.js)

Проекты хранятся в отдельном ключе `legal_ocr_projects` в localStorage (макс. 30). Структура проекта:
```json
{
  "id": "proj_...",
  "title": "Дело № 123/2026",
  "description": "Описание дела",
  "documentIds": ["doc_1", "doc_2", "doc_3"],
  "sharedPD": { "persons": [], "otherPD": [] },
  "createdAt": "...",
  "updatedAt": "..."
}
```

Документы проекта хранятся в общей истории (`legal_ocr_history`), но имеют поле `projectId` — фильтруются из общей истории на главной (`freeHistory`).

### CRUD операции (history.js)

- `loadProjects()`, `saveProject()`, `deleteProject()`, `getProject()`
- `createProject(title, description)`
- `addDocumentToProject(projectId, docId)`
- `removeDocumentFromProject(projectId, docId)`
- `updateProjectSharedPD(projectId, sharedPD)`

### Добавление документов в проект

Три способа:

1. **Распознать PDF** — `handleProjectRecognize()`: OCR → анализ ПД с `existingPD` из последнего документа проекта → `mergePD()` → `assignLetters()` → автосохранение с `projectId`
2. **Перенести из истории** — `handleAddDocFromHistory()`: берёт документ из общей истории → `mergeDocIntoProject()` → устанавливает `projectId` (документ исчезает из общей истории)
3. **Загрузить .юрдок** — `handleProjectImport()`: импортирует → `mergeDocIntoProject()`

### mergeDocIntoProject — ключевая функция

При добавлении документа в проект:
1. Если `editedHtml` существует — используется как основа (сохраняет все правки пользователя)
2. Новые ПД из проектной базы ищутся в текстовых узлах DOM через regex
3. Найденные — оборачиваются в `<mark>` маркеры
4. Документ получает `projectId` и сохраняется

### Сквозная база ПД

- `mergePD(existingPD, newPD)` — мёржит два набора ПД: лица по `fullName`, otherPD по `type + value` (нормализация пробелов)
- `assignLetters(pd, existingPD)` — нумерация букв продолжается с того места, где остановилась предыдущая часть
- `analyzePD(..., existingPD)` — передаёт существующую базу в промпт: API не дублирует известных, только добавляет новых

### Визуализация ПД в контексте проекта

Когда документ открыт из проекта, панель ПД показывает полную накопленную базу:
- ПД с маркерами в тексте — нормальный вид, навигация стрелками
- ПД без маркеров (из предыдущих частей) — `opacity: 0.4`, подпись «нет в документе», стрелки и клик отключены
- Управляется через `pdIdsInDoc` (Set) и хелпер `pdInDoc(id)`

### Итоговый документ проекта

`buildProjectSummary()` собирает все документы проекта в один:
1. Склеивает `editedHtml` всех частей с разделителями `<div class="part-separator">` между ними
2. Берёт `personalData` из последнего документа (накопительная)
3. Мёржит `anonymized` из всех документов
4. Сохраняет как документ с `isProjectSummary: true`

Разделители частей (`part-separator`) визуально отличаются от разделителей страниц (`page-separator`): синяя линия вместо серой, надпись «Часть N: название». Оба типа подсвечиваются при экспорте и предлагаются к удалению.

При пересборке — модалка подтверждения, старый итоговый документ заменяется.

`exportDocument(entry)` — скачивает JSON-файл с расширением `.юрдок`.

### Импорт
`importDocument(file)` — читает `.юрдок` файл, валидирует, сохраняет в localStorage с новым id. При импорте в проект — дополнительно вызывается `mergeDocIntoProject()`.

## Двухпроходная аннотация ПД (RichEditor.js)

### Первый проход — построчно через annotLine

`buildAnnotatedHtml()` разбивает текст по `\n`, каждую строку обрабатывает через `annotLine()`:
- **Persons** — `buildPersonPattern()`: гибкий regex с падежами, инициалами, регистром первой буквы
- **OtherPD** — `buildOtherPdPattern()`: разбивает value по пробелам → соединяет через `[\s\n]+`, флаг `gi`

### Второй проход — DOM-поиск непомеченных otherPD

После построения HTML парсит его в DOM и ищет otherPD без маркеров через текстовые узлы. Находит значения разбитые на две строки или содержащие `<mark class="uncertain">` внутри.

### Переаннотация после снятия uncertain mark

`onUncertainResolved` → `handleUncertainResolved()`:
1. `node.parentNode.normalize()` — склеивает текстовые узлы
2. Сканирует DOM редактора, находит otherPD без маркеров → ставит маркеры
3. Обновляет HTML и snapshot

## Автосохранение

Документ сохраняется автоматически сразу после распознавания. `lastSavedState` устанавливается синхронно. Дополнительно: `useEffect` по `view === VIEW_RESULT` через 300мс обновляет `lastSavedState` из реального DOM (компенсация нормализации HTML браузером), чтобы `isDirty()` не давал ложных срабатываний.

---

## Обезличивание

Реализовано через **DOM-патчинг** (`patchPdMarks` в RichEditor.js), а не пересборкой HTML. Это сохраняет пользовательские правки в редакторе.

**Важно:** `pdRef` и `anonRef` обновляются синхронно прямо в обработчиках (до `setAnonymized`/`setPersonalData`), чтобы `snap()` всегда читал актуальное состояние.

---

## Система undo/redo

Стек хранит **результирующие снимки** `{ html, pd, anon }`. `replaceTopSnap` заменяет верхний элемент — используется когда DOM уже изменён notifyChange, но pd обновился позже.

---

## Панель ПД

Три группы + легенда (частные лица, профучастники, другие ПД, неточное распознавание). Навигация ↑↓ с `visibility: hidden/visible` — панель не скачет при hover. При навигации scrollTop панели сохраняется и восстанавливается.

Записи ПД **не удаляются автоматически** из панели — только по явному действию пользователя (ПКМ → «Не является ПД»).

## Предстоящие задачи

### Задача 1 — Автоматическое разбиение большого PDF в проекте

При загрузке большого PDF (>20 страниц) в проект — автоматически нарезать на части по ~20 страниц и обрабатывать последовательно без участия пользователя:

1. Пользователь загружает один большой PDF в проект
2. Приложение определяет количество страниц
3. Автоматически разбивает на части (~20 страниц каждая)
4. Каждая часть распознаётся последовательно с прогрессом
5. Сквозная база ПД передаётся от части к части через `existingPD`
6. Каждая часть сохраняется как отдельный документ проекта
7. После обработки всех частей — автоматически собирается итоговый документ через `buildProjectSummary()`

**Технические соображения:**
- `pdfToImages()` уже поддерживает постраничную конвертацию
- Нужен новый `handleProjectRecognizeLarge()` с циклом по batch-ам
- Прогресс-бар должен показывать общий прогресс (часть X из Y, страница Z из W)
- При ошибке на одной части — предложить продолжить или остановиться
- `recognizeDocument()` и `analyzePD()` уже поддерживают `existingPD`

---

### Задача 2 — Экспорт PDF с оригинальной визуализацией

**Проблема:** Текущий PDF-экспорт генерирует цифровой документ из HTML. Оригинальные сканы (подписи, печати, цвет бумаги, рукописные пометки) теряются.

**Задумка:** После обезличивания в редакторе — вносить изменения обратно в оригинальные изображения страниц. Итоговый PDF = оригинальные страницы-картинки с изменёнными текстовыми фрагментами.

**Технический подход (для проработки):**

1. Оригинальные страницы хранятся в `originalImages` (base64 PNG/JPEG) пока документ открыт — это база для наложения изменений.

2. Сопоставление текста с координатами на странице — варианты:
   - Сохранять bounding boxes из OCR (Claude возвращает координаты через `tool_use`)
   - Передавать оригинальное изображение + список замен обратно в Claude с просьбой указать координаты

3. Для каждой изменённой страницы: через Canvas API накладывать белые прямоугольники на ПД-фрагменты и рисовать новый текст поверх.

4. Собирать итоговый PDF из модифицированных изображений через `jsPDF` или аналог.

**Сложности:**
- Точное сопоставление координат OCR с позицией на изображении
- Подбор шрифта и размера под оригинал
- Альтернатива: повторный запрос к Claude — «вот изображение страницы, вот список замен — укажи координаты для каждой замены»
