# ЮрДок — OCR и обезличивание юридических документов

Веб-приложение для распознавания юридических документов и обезличивания персональных данных. Поддерживает OCR для сканов/PDF, прямой разбор DOCX и быстрый режим вставки цифрового текста для ручных проверок.

Этот `README` приведён в состояние handoff-документа: по нему можно продолжать работу над приложением в новом чате без восстановления длинного контекста из истории.

## Что умеет приложение сейчас

- Загружать `PDF`, `JPG`, `PNG`, `DOCX`
- Принимать вставленный цифровой текст через отдельную вкладку `Текст`
- Распознавать текст через `Claude`, `OpenAI` или `Gemini`
- Находить персональные данные и размечать их в редакторе
- Поддерживать проекты с общей сквозной базой ПД по нескольким частям дела
- Показывать OCR-ошибки отдельно от неоднозначных упоминаний лиц
- Экспортировать результат в `PDF`, `DOCX` и собственный формат `.юрдок`

## Технологии

- Frontend: `React 18`, `Create React App`
- OCR / LLM API: `Claude`, `OpenAI`, `Gemini`
- PDF: `pdfjs-dist`
- DOCX parsing: ручной разбор OOXML через `JSZip`
- Хранение: `localStorage`

## Основной пользовательский поток

### 1. Главная страница

Под полем API-ключа есть две вкладки:

- `Документы` — стандартная загрузка `PDF/JPG/PNG/DOCX`
- `Текст` — вставка цифрового текста для быстрого обезличивания и ручных проверок

По умолчанию активна вкладка `Документы`.

### 2. Распознавание

#### PDF / изображения

1. `src/utils/pdfUtils.js` конвертирует страницы в изображения
2. `src/utils/claudeApi.js -> recognizeDocument()` делает OCR
3. Выполняется quality-check OCR
4. Выполняется анализ ПД
5. `src/components/RichEditor.js -> buildAnnotatedHtml()` строит разметку

#### DOCX

1. `src/utils/docxParser.js` извлекает текст без OCR
2. Текст сразу идёт в анализ ПД

#### Вставленный текст

1. Пользователь вставляет текст во вкладке `Текст`
2. `src/utils/claudeApi.js -> analyzePastedText()` анализирует его напрямую
3. Документ сохраняется как источник `paste`

## Текущая модель данных ПД

`personalData` сейчас имеет вид:

```json
{
  "persons": [],
  "otherPD": [],
  "ambiguousPersons": []
}
```

### `persons`

Обычные лица с полями наподобие:

- `id`
- `fullName`
- `label`
- `mentions[]`
- `role`

### `otherPD`

Другие персональные данные:

- телефоны
- адреса
- даты рождения
- документы
- прочее

### `ambiguousPersons`

Новая категория для читаемого текста, где проблема не в OCR, а в неоднозначной идентификации лица.

Типовой пример:

- в тексте уже есть `Иванов Вячеслав Александрович`
- позже встречается `Вячеслав`
- но в документе есть и другой `Вячеслав`

В таком случае фрагмент должен идти не в `persons[].mentions`, а в `ambiguousPersons`.

## Визуальная разметка в редакторе

Сейчас в редакторе есть три смысловые категории пометок:

- обычные ПД — стандартные фиолетовые маркеры
- OCR-warning — оранжевая пометка `неточное распознавание`
- ambiguous person — отдельная синяя пометка `неоднозначное упоминание лица`

Важно:

- `OCR-warning` используется только для проблем распознавания текста
- `ambiguousPersons` используется только для неоднозначной идентификации человека
- эти категории специально разведены и не должны смешиваться

Легенда в интерфейсе уже обновлена и содержит категорию `неоднозначное упоминание лица`.

## Что сделано в последних изменениях

### 1. Исправлен захват последней буквы предыдущего предложения

Была системная ошибка:

- `... ничего не пояснил. Полуянович И.В. ...`
- маркер ошибочно захватывал `л. Полуянович И.В.`

Исправление:

- в `src/components/RichEditor.js` в `buildPersonPattern()` добавлена жёсткая левая граница
- паттерн больше не стартует с последней буквы предыдущего слова

### 2. Сохранена поддержка OCR-ошибок по регистру

Поддерживаются варианты:

- `Баданов А.С.`
- `баданов А.С.`
- `А.С. Баданов`
- `а.С. Баданов`
- `а.с. Баданов`
- `А. С. Баданов`
- `а. С. Баданов`
- `а. с. Баданов`

То же касается фамилий вида:

- `З.С. Лебедев`
- `з.С. Лебедев`
- `з.с. Лебедев`
- `З. С. Лебедев`
- `з. С. Лебедев`
- `з. с. Лебедев`

### 3. Добавлена поддержка кратких имён

В промпт анализа ПД добавлено правило:

- если короткое имя явно относится к уже названному человеку, оно должно попадать в `mentions[]`
- если короткое имя неоднозначно, оно не должно автоматически привязываться

Пример ожидаемого поведения:

- `Иванов Вячеслав Александрович ... Слава ...` -> `Слава` идёт в `mentions[]`
- если в документе несколько подходящих `Вячеслав`, спорное `Вячеслав` должно идти в `ambiguousPersons`

### 4. Добавлена категория `ambiguousPersons`

Изменения уже внесены в:

- промпт `src/utils/claudeApi.js`
- редактор `src/components/RichEditor.js`
- стили `src/components/RichEditor.css`
- легенду и обработчики `src/App.js`

### 5. Добавлено контекстное меню для `ambiguousPersons`

Для синей пометки доступны действия:

- привязать к существующему лицу
- создать новое лицо
- снять пометку

### 6. Исправлен undo/redo для ручной работы с ambiguous-метками

Логика снимков в `src/App.js` была выровнена:

- удалён старый `skipNextHtmlSnapRef`
- `handleEditorHtmlChange()` всегда пишет один консистентный snapshot
- attach/add действия больше не создают лишний промежуточный шаг undo

## Важная особенность `ambiguousPersons`

Модель может вернуть противоречивый ответ:

- и в `ambiguousPersons`
- и одновременно в `persons[].mentions`

Чтобы UI не терял синюю пометку, в `src/utils/claudeApi.js` добавлена постобработка:

- все значения из `ambiguousPersons` удаляются из `persons[].mentions`

Это не теоретическая защита. Она нужна реально: на живых запросах модель иногда возвращает конфликтующие данные.

Текущее состояние:

## Ограничения, которые важно помнить

### 1. Анализ ПД ограничен первыми 25 000 символов

В `src/utils/claudeApi.js` сейчас есть ограничение:

- для PD analysis берутся только первые `25000` символов текста

Следствие:

- в длинном документе хвост текста может не попасть в анализ ПД

### 2. Long document warning

В `src/App.js` показывается предупреждение, если текст длиннее `50000` символов.

### 3. Большие PDF

`src/utils/pdfUtils.js` сжимает страницы, чтобы base64 оставался в разумных пределах.

Текущее поведение:

- жёсткого лимита по `file.size` на входе нет
- реальные ограничения идут через:
  - число страниц
  - разрешение страниц
  - время рендера в браузере
  - размер base64-данных
  - длину итогового распознанного текста

### 4. Хранилище

Все документы и проекты живут в `localStorage`.

Следствие:

- нет серверного хранения
- очистка браузерных данных удаляет историю
- при большом объёме можно упереться в квоту
