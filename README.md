# ЮрДок — Распознавание и обезличивание юридических документов

Веб-приложение для OCR-распознавания юридических документов и автоматического обезличивания персональных данных. Пользователь загружает сканы (JPG/PNG/PDF) или DOCX, приложение распознаёт текст через Claude/OpenAI/Gemini API, находит персональные данные и позволяет заменить их на метки (А., Б., [ФИО 1] и т.д.).

## Архитектура и поток данных

### Режимы загрузки

**Изображения/PDF:**
1. `pdfUtils.js` конвертирует PDF в base64-изображения постранично
2. `claudeApi.js → recognizeDocument()` — OCR каждой страницы отдельным запросом
3. Страницы склеиваются с маркерами `[PAGE:N]`
4. Quality check — поиск OCR-артефактов и семантических ошибок
5. Анализ ПД — структурированный JSON с persons и otherPD
6. `RichEditor.js → buildAnnotatedHtml()` — вставка `<mark data-pd-id="...">` в HTML

**DOCX:**
1. `docxParser.js` читает XML напрямую через JSZip (без OCR)
2. Сразу идёт на анализ ПД (quality check пропускается)

### Состояние в App.js

Ключевые useState:
- `view` — `'home' | 'processing' | 'result'`
- `personalData` — `{ persons: [...], otherPD: [...] }` — данные от API
- `anonymized` — `{ [id]: bool }` — какие ПД скрыты
- `editorHtml` — текущий HTML редактора
- `pdWidth`, `viewerWidth` — ширины панелей (изменяются ресайзером)
- `pdNavState` — `{ [id]: { cur, total } }` — состояние навигации по упоминаниям

Ключевые useRef (не вызывают ре-рендер):
- `editorDomRef` — прямая ссылка на DOM редактора
- `pdRef` — синхронная копия `personalData` (всегда актуальна)
- `anonRef` — синхронная копия `anonymized` (всегда актуальна)
- `undoStackRef` — стек снимков для undo/redo `[{html, pd, anon}]`
- `undoIndexRef` — текущая позиция в стеке
- `pdCleanupTimerRef` — дебаунс-таймер очистки панели ПД

### Обезличивание

Реализовано через **DOM-патчинг** (`patchPdMarks` в RichEditor.js), а не пересборкой HTML. Это сохраняет пользовательские правки в редакторе.

При клике на элемент ПД → `handlePdClick(id)` → вычисляет `nextAnon` синхронно → `patchPdMarks(...)` → `setEditorHtml(...)` → `setAnonymized(...)` → `pushSnap(...)`.

**Важно:** `pdRef` и `anonRef` обновляются синхронно прямо в обработчиках (до `setAnonymized`/`setPersonalData`), чтобы `snap()` всегда читал актуальное состояние.

## Система undo/redo

Единая история для всех действий — и ПД-манипуляций, и текстовых изменений.

### Принцип

Стек хранит **результирующие снимки** (состояние после действия), а не «состояние до». Это позволяет корректно отменять последовательные быстрые действия:
- `stack[0]` — начальное состояние документа
- `stack[1]` — после первого действия
- `stack[N]` — после N-го действия
- Ctrl+Z: `index--`, восстанавливается `stack[index]`
- Ctrl+Shift+Z / Ctrl+Y: `index++`

### Снимок `{ html, pd, anon }`
- `html` — `editorDomRef.current.innerHTML` в момент снимка
- `pd` — `pdRef.current` (personalData)
- `anon` — `anonRef.current` (anonymized)

### Когда делается снимок

- **`handlePdClick`** — `pushSnap` после `patchPdMarks` и `setEditorHtml`
- **`anonymizeAllByCategory`** — `pushSnap` после всех DOM-изменений
- **`handleRemovePdMark`** — `replaceTopSnap` (notifyChange уже запушил HTML, нужно обновить pd)
- **`handleAttachPdMark`** — `replaceTopSnap` после DOM-коррекции класса маркера
- **`handleAddPdMark`** — `replaceTopSnap` после DOM-коррекции класса и реального id
- **`removeUncertainMark` / `applyUncertainSuggestion`** — `notifyChange()` → `handleEditorHtmlChange` → `pushSnap`
- **Текстовый ввод** — каждый `notifyChange()` → `pushSnap` немедленно (дебаунс убран намеренно, т.к. текстовый ввод в редакторе редкий)

### `replaceTopSnap` vs `pushSnap`

`replaceTopSnap` заменяет верхний элемент стека — используется когда DOM уже изменён и `notifyChange` создал снимок, но затем `setPersonalData` обновил `pd`. Без этого снимок содержал бы старый `pd`.

### Клавиши

Перехватываются глобально на `document` в capture-фазе. Работает на любой раскладке (`e.code` не зависит от языка). Не перехватывается в `INPUT`, `SELECT`, `TEXTAREA`.

- **Undo:** Ctrl+Z / Cmd+Z
- **Redo:** Ctrl+Shift+Z / Cmd+Shift+Z и Ctrl+Y / Cmd+Y

## Маркеры ПД в редакторе

### CSS-классы маркеров

```
mark.pd.priv       — частное лицо (фиолетовый)
mark.pd.prof       — профучастник (зелёный)
mark.pd.oth        — другое ПД (бежевый)
mark.pd.anon       — скрытое ПД (залитый цвет)
```

Все маркеры имеют `contenteditable="false"` — курсор не может войти внутрь.

### data-атрибуты маркера

- `data-pd-id` — id записи в personalData
- `data-original` — исходный текст документа (для восстановления при показе/undo)

### Взаимодействие с маркерами

- **Левый клик** — скрыть/показать ПД
- **Правый клик на маркере** — «Не является ПД»
- **Правый клик на выделенном тексте** — привязать к существующему ПД или добавить новое
- **Backspace/Delete** рядом с маркером — удалить маркер и текст целиком
- Выделение частично захватывает маркер — автоматически расширяется до полных границ

### Обозначения частных лиц

`ALPHA_PRIVATE` — русский алфавит А.–Я. (29 букв). При исчерпании: Л-30, Л-31, ...

## Промпты (claudeApi.js)

Три промпта выполняются последовательно:

### SYS_OCR
System-промпт для распознавания. Задаёт формат:
- `## НАЗВАНИЕ` — заголовок документа
- `[CENTER]текст[/CENTER]` — центрированный текст
- `[LEFTRIGHT: левый | правый]` — строки с текстом слева и справа
- `[RIGHT-BLOCK]текст` — блок прижатый вправо (шапка)
- `[INDENT]текст` — абзац с отступом
- `⚠️[НЕТОЧНО: слово]` или `⚠️[НЕТОЧНО: слово | правильный_вариант]` — неточное распознавание
- `⚠️[НЕЧИТАЕМО]` — нечитаемый фрагмент

### PROMPT_QUALITY
Ищет три типа ошибок OCR: грамматическую невозможность, логическое несоответствие, нарушение юридического шаблона.

### PROMPT_PD
Анализирует текст и возвращает JSON с `persons` и `otherPD`. Ключевые правила:
- `fullName` всегда в формате «Фамилия И.О.»
- `professional` — составляет/подписывает/утверждает/ведёт дело
- `private` — потерпевший, свидетель, заявитель, понятой
- OCR-артефакты («г р.», «г. р.») добавляются в `mentions[]`

## Логика buildPersonPattern (RichEditor.js)

Строит regex для поиска упоминания человека с учётом падежных окончаний, инициалов и регистра первой буквы (для OCR-артефактов с маленькой буквы).

## Панель персональных данных

Три группы: Частные лица / Профучастники / Другие ПД. Каждый элемент:

```
[А.]  Корчагова Е.В.     [↑][2/5][↓]  👁    ← строка 1
      заявитель                               ← строка 2 (роль)
```

- Навигация ↑↓ через `initNavCounter` + `navigateToPd`, flash через `IntersectionObserver`
- Скролл панели изолирован через `wheel` listener с `preventDefault()`
- При исчезновении всех маркеров — запись удаляется из панели через 1 сек (дебаунс)

## Предстоящие задачи

### Задача 1 — Большие документы и надёжное хранение истории

**А. Работа с многостраничными делами (тысячи страниц)**

Сейчас за один раз обрабатывается ~20 страниц. Для больших дел нужно:

1. **Сквозная база ПД между частями.** При загрузке следующей части передавать накопленный `personalData`. Новые лица получают следующие свободные буквы, уже известные — свои прежние метки. Требует обновления `PROMPT_PD`: передавать существующий словарь лиц, чтобы модель не создавала дубликаты.

2. **Автоматическое разбиение большого файла.** При загрузке PDF/DOCX большого объёма — автоматически нарезать на части (~20 страниц), обрабатывать последовательно с прогрессом. Каждая часть сохраняется в историю отдельно, но связана через `caseId`.

3. **Группировка в «Дело».** Концепция группы документов с общей базой ПД — в истории отображается как раскрывающийся список. Или более простой вариант: флаг «продолжить с базой ПД из...» при загрузке нового документа.

**Б. Надёжное хранение истории**

История в localStorage теряется при сбросе данных браузера. Нужно:

1. **Экспорт документа** — кнопка в карточке истории → скачать `.юрдок`-файл (JSON с `editedHtml`, `personalData`, `anonymized`, `rawText`, метаданными).

2. **Импорт документа** — загрузить `.юрдок`-файл → он появляется в истории и открывается в редакторе в сохранённом виде.

Формат `.юрдок` — обычный JSON, переименованный для узнаваемости.

### Задача 2 — Экспорт PDF с оригинальной визуализацией

**Проблема:** Текущий PDF-экспорт генерирует цифровой документ из HTML. Оригинальные сканы (подписи, печати, цвет бумаги, рукописные пометки) теряются.

**Задумка:** После обезличивания в редакторе — вносить изменения обратно в оригинальные изображения страниц. Итоговый PDF = оригинальные страницы-картинки с изменёнными текстовыми фрагментами.

**Технический подход (для проработки):**

1. Оригинальные страницы хранятся в `originalImages` (base64 PNG/JPEG) пока документ открыт — это база для наложения изменений.

2. Сопоставление текста с координатами на странице — варианты:
   - Сохранять bounding boxes из OCR (Claude возвращает координаты через `tool_use`)
   - Передавать оригинальное изображение + список замен обратно в Claude с просьбой отрисовать изменения

3. Для каждой изменённой страницы: через Canvas API накладывать белые прямоугольники на ПД-фрагменты и рисовать новый текст поверх.

4. Собирать итоговый PDF из модифицированных изображений через `jsPDF` или аналог.

**Сложности:**
- Точное сопоставление координат OCR с позицией на изображении
- Подбор шрифта и размера под оригинал
- Альтернатива: повторный запрос к Claude — «вот изображение страницы, вот список замен — нарисуй изменения»
